# Claims register

Every user-facing security claim, graded **TRUE**, **TRUE-WITH-CAVEATS**
(caveats named), or **NOT-MADE**. A claim may not appear in README,
PRIVACY, UI copy, or marketing unless it appears here with a grade and an
enforcing reference. If code and this register disagree, one of them is a
bug — same rule as PROTOCOL.md.

Permanently NOT-MADE, never to appear anywhere: "uncrackable",
"military-grade", "no metadata", "messages never leave your devices".

| Claim | Grade | Enforced by |
|-------|-------|-------------|
| Every message, draft, sync page, and media chunk on the wire is AES-256-GCM ciphertext under link-derived keys | TRUE | `src/lib/crypto/envelope.ts`, `keys.ts` (HKDF subkeys, §1.2); E2E wire assertion hooks `RTCDataChannel.send` and the relay frames (`required-typing.spec.ts`, `required-network.spec.ts`) |
| The link key never reaches any server | TRUE-WITH-CAVEATS — the browser can leak the URL: history sync to a non-E2E cloud account, share-sheet targets | URL-fragment design (§1.1; fragments are not sent in HTTP); caveat disclosed in PRIVACY.md and the share copy |
| Members cannot forge, edit, or delete each other's messages or reactions | TRUE-WITH-CAVEATS — drafts/presence are unsigned and spoofable by room members (§2); history served during sync (deleted-state, reaction maps) is trusted from the serving member (§3.5/§3.7); ECDSA is quantum-forgeable, so a link-holding adversary with a live cryptographically-relevant quantum computer could impersonate members and minters until key rotation — an attribution break, not a confidentiality one, with no harvest-now exposure; deliberate posture + rotation trigger in PROTOCOL.md §1.3 | ECDSA P-256 envelope signatures (§2); fail-closed ownership checks (`src/lib/p2p/ownership.ts`, `tests/unit/ownership.test.ts`); reaction membership keyed to the envelope-verified deviceId (`src/lib/p2p/reactions.ts`, `tests/unit/reactions.test.ts`) |
| Chat history and media are encrypted at rest | TRUE-WITH-CAVEATS — same-origin script (XSS, malicious extension) can use the non-extractable keys in place (§6); legacy plaintext for rooms whose keys were never persisted is deleted, not encrypted (§4) | `src/lib/storage/messages.ts`, `src/lib/media/blob-store.ts`; `message-storage.test.ts` (ciphertext-at-rest, AAD fail-closed, migration) |
| Leaving with burn removes the room's keys, history, and media from the device | TRUE-WITH-CAVEATS — peers keep their copies; the device keypair is per-device and survives; a message landing in another tab in the instant before the burn broadcast arrives can re-persist | `src/lib/storage/burn.ts` (+ BroadcastChannel evacuation); `burn.test.ts`; residue-free E2E in `room-page.spec.ts` |
| Live drafts, history sync, and media never transit the server | TRUE | `relayEligible` + inbound policy in `src/lib/p2p/connection.ts`; `relay-policy.test.ts`; E2E asserts zero `eph` envelopes reach the relay |
| Relay use is visible in the UI | TRUE | `ConnectionStatus.svelte` ("relayed via server", per-peer chips); asserted in `required-network.spec.ts` |
| The signaling operator cannot read or inject messages | TRUE-WITH-CAVEATS — it sees rendezvous metadata, relayed-SDP IPs, and on the relay path the sender deviceId + ciphertext size + timing (PRIVACY.md) | HMAC signaling auth (§3.1), envelope signatures + replay guard (§2), hello channel binding (§3.4) |
| Photos are stripped of EXIF/GPS before sending | TRUE | unconditional re-encode in `src/lib/media/image.ts` (§5.3); EXIF E2E test |
| Forward secrecy (wire-capture, scoped) | TRUE-WITH-CAVEATS — the §1.4 generation ratchet is live: rooms rotate to fresh random generation secrets on joins and leaves, so relay-archived ciphertext of generations ≥1 stays unreadable to a passive operator even given a later-leaked link, and retired generations' keys are destroyed two ratchets on. Since v4, generation secrets transit only **hybrid-wrapped per recipient device** (X-Wing: X25519 + ML-KEM-768), so a grant captured at an endpoint or recovered from recorded DTLS by a future quantum attacker stays unreadable without that device's key store. Named exclusions, which must ride any user-facing statement: a leaked link still grants *entry* + history-by-sync (capability model, visible as a joining peer); at-rest history stays under static `k_storage` (§4); compromise of a recipient device's key store opens grants addressed to it (and its at-rest history regardless); an active link-holder at grant time is simply a member; relay-only peers cannot receive rotations (stranded, shown in UI); a member's own ratchet griefing is availability-only (§1.4/§6); the PQC implementation (`@noble/post-quantum`, pure JS, exact-pinned) is FIPS-203/204-final and Wycheproof/ACVP-tested but **not independently audited** — no independently audited pure-JS PQC exists (2026-06). Never stated unqualified | PROTOCOL.md §1.3/§1.4 v4; `src/lib/crypto/kem.ts` (X-Wing wrap, AAD-bound), `src/lib/p2p/ratchet.ts` + `crypto-session.ts` (engine, trial decrypt, per-recipient wrapped grants, direct-only); `tests/unit/kem.test.ts`, `ratchet-engine.test.ts`, `session.test.ts` (§7 v4 rows); `required-ratchet.spec.ts` (join→rotation E2E, relay stranding); scoped wording in README "Honest limits" |
| Post-quantum content confidentiality | TRUE-WITH-CAVEATS — content keys are symmetric-only (Grover leaves ~128-bit effective strength), and since v4 the one asymmetric path near content — generation-secret distribution — is hybrid post-quantum (X-Wing), closing harvest-now-decrypt-later on grants. ECDSA signatures remain quantum-forgeable — an authenticity risk against link holders only, not confidentiality (Phase-2 item 9: documented posture with rotation trigger); the DTLS transport layer is classical ECDHE in most engines (Chrome 142+ is hybrid-PQ) but only wraps already-E2E ciphertext and is browser-controlled | Key schedule PROTOCOL.md §1, §1.4 Quantum note (v4), threat model §6 |
| No accounts, analytics, tracking, third-party scripts, or runtime CDN | TRUE | CSP in `svelte.config.js` (self + hashed inlines); `connect-src` pinned to self + the build-time signaling origin (no scheme-wide `ws:`); self-hosted fonts; zero third-party runtime origins |
| You don't have to trust the server with content | TRUE-WITH-CAVEATS — you re-trust the *served JavaScript* on every page load; a malicious build could exfiltrate keys. This is the ceiling of every web-delivered messenger vs an installed, signed app. SRI does not raise it (the adversary serves the entry HTML); genuine paths are reproducible-build digests and a pinned verifier | E2E encryption design; ceiling disclosed in README "Honest limits"; full analysis in `docs/BUNDLE_VERIFICATION.md` |
